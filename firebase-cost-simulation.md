# Firebase 費用シミュレーション（FMS 大規模大会向け）

最終更新：2026-05-24
対象：選手200名 + 観戦者400名規模の大会で FMS（フェンシングマネージャーシステム）を運用する場合

---

## 1. Firebase のプラン体系

Firebase には **2つだけ** プランがあります：

| プラン | 料金 | 説明 |
|---|---|---|
| **Spark** | 無料 | 無料枠まで使えて、超えると停止 |
| **Blaze** | 従量課金 | 無料枠は同じ。超えた分だけ課金 |

> **たとえ話**：Spark は「使い放題の格安SIM（容量超えたら通信停止）」、Blaze は「使った分だけ請求の段階制プラン（容量超えたら自動で続く代わりに従量課金）」と思ってください。

---

## 2. 無料枠（Spark/Blaze 共通）

Firestore（データベース）の1日あたりの無料枠：

| 項目 | 1日の無料枠 |
|---|---|
| 読み取り（reads） | 50,000 回 |
| 書き込み（writes） | 20,000 回 |
| 削除（deletes） | 20,000 回 |
| 保存容量 | 1 GB |
| 通信量 | 10 GB/月 |

---

## 3. Blaze プランの追加料金（超過分のみ）

| 項目 | 単価 |
|---|---|
| 読み取り | $0.18 / 100,000 回（≒ 27円） |
| 書き込み | $0.18 / 100,000 回（≒ 27円） |
| 削除 | $0.02 / 100,000 回（≒ 3円） |
| 保存容量 | $0.026 / GB/月（≒ 4円） |

※1ドル＝150円換算

## 4. ご質問の前提条件で試算

- 選手 **200名**
- 観戦者（保護者含む） **400名**
- 合計アクセス者 **600名**
- 大会1日6時間
- FMS では **リロード方式** に変更したため、画面を開いた瞬間と「更新ボタン」を押した時だけ読み取りが発生

### シナリオA：FMS の標準動作（リロード方式・観戦者は手動更新）

- 観戦者は試合の節目ごとに1〜2回 / 1時間程度更新する想定 → 6時間で約 **8回**
- 運営は10秒に1回スコア更新（書き込み） → 1試合中 約 **30回** の書き込み
- 1ページのリロード = 約 **1〜3 文書の読み取り**（大会データが1ドキュメントなので軽量）

#### 1日あたりの読み書き計算

| 項目 | 計算 | 回数 |
|---|---|---|
| 観戦者の読み取り | 600人 × 8回更新 × 3文書 | 14,400 |
| 運営の読み取り | 5人 × 30回 × 2文書 | 300 |
| 運営の書き込み | 200名 × 約25試合 × 1回 | 5,000 |
| **合計 読み取り** |  | **約 15,000 回** |
| **合計 書き込み** |  | **約 5,000 回** |

→ **1日の無料枠（50,000読み取り / 20,000書き込み）以内に余裕で収まる**

→ **このシナリオでは 0円（無料） で運用可能**

### シナリオB：リアルタイム接続（旧仕様・全員が常時接続）

参考：従来の `subscribeTournament` を使うと、データ更新のたびに全員に再配信されます。

- 600人 × 6時間 × 60秒 × 0.5（平均更新頻度を仮定） = 約 **108万回 / 日 の読み取り**

| シナリオ | 読み取り回数 | 費用（読み取り） |
|---|---|---|
| 軽め（1更新=10文書） | 約 108万回 | 約 $1.9（≒ 290円） |
| 中（1更新=20文書） | 約 216万回 | 約 $3.9（≒ 590円） |
| 重い（1更新=30文書） | 約 324万回 | 約 $5.8（≒ 870円） |

→ **1大会あたり 数百円〜1,000円程度**

### シナリオC：観戦者の自動更新3分間隔（FMS の自動更新ON）

- 600人 × 6時間 × 20回（3分毎）× 3文書 = 約 **216,000 回 / 日**
- 無料枠超過：216,000 - 50,000 = **166,000 回**
- 追加費用：166,000 / 100,000 × $0.18 = **約 $0.3（≒ 45円）**

→ **1大会あたり 数十円〜100円程度**

## 5. 結論

| 運用方式 | 1大会の費用 | おすすめ度 |
|---|---|---|
| 手動更新のみ（FMS標準） | **0円** | ⭐⭐⭐ 強くおすすめ |
| 3分自動更新 | **約45円** | ⭐⭐ コスパ良好 |
| リアルタイム接続（旧仕様） | **300〜1,000円** | ⭐ 高頻度試合のみ |

**月1〜2大会の運用なら、年間費用は最大でも 数千円程度** で済む見込みです。

## 6. 想定外の費用増を避けるための対策

### 対策1：Firebase の予算アラートを設定
1. [Google Cloud Console](https://console.cloud.google.com/) → 該当プロジェクト
2. お支払い → 予算とアラート → 「予算を作成」
3. 月予算を設定（例：1,000円）
4. 50% / 90% / 100% でメール通知が来るように設定

### 対策2：Firestore セキュリティルール
書き込みを運営者だけに制限することで、悪意ある外部アクセスを防止。

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /tournaments/{id} {
      allow read: if true;   // 観戦は誰でもOK
      allow write: if false; // 書き込みは Firebase Auth で制限する場合のみ true 化
    }
  }
}
```

### 対策3：自動更新間隔を伸ばす
FMS の観戦ページで「自動更新3分毎」を切れば、観戦者は手動更新のみになり、読み取りがほぼゼロに。

---

## 7. 注意点（推測含む）

- 推測：**今回の試算は「1大会データ = 1 Firestore ドキュメント」の前提**です。実際には複数ドキュメントに分けることもあるため、上記より若干増える可能性あり。
- 推測：**実際の費用は前月の使用量を翌月初に請求**されます。当日の急なスパイクで請求が確定するわけではありません。
- 確実：**Spark プラン（無料）のままでも、無料枠を超えた瞬間にサービス停止**するだけで、勝手に課金されることはありません。心配なら Spark のまま運用可能です。
